Algorithm & Implementation Details

Core PageRank Formula

All three implementations use the same damped PageRank recurrence:

new_rank(v) = 0.15 + 0.85 × SUM rank(u) / outDeg(u)   for each u → v

Same Iterative Algorithm

All three implementations follow the identical algorithm: (1) initialize rank=1 for pages with outgoing links, (2) join ranks with edges, divide by out-degree, aggregate, (3) apply damping formula, repeat for 10 iterations. Pages that receive no contributions in an iteration are dropped, causing active page count to shrink from 680K to ~597K.

Why GraphFrames Version Uses Same Algorithm

Custom Algorithm, Not Built-in pageRank()

The GraphFrames implementation does not use GraphFrames' built-in pageRank() method. Instead, it uses from graphframes import GraphFrame to construct the graph and then runs the same manual iterative JOIN-aggregate loop as Tasks 1-2. This is why all three implementations produce byte-identical results (same page IDs, same ranks, same iteration counts).

Why DuckDB Is 39× Faster

No JVM Overhead • No Serialization • Columnar Processing

1. No JVM overhead: Spark must start a JVM, load classes, and manage GC — even before the first query. DuckDB is a lightweight in-process library.

2. No serialization: Spark serializes data between JVM and Python via Py4J/socket for every action (.count(), .collect()). DuckDB operates entirely in-process.

3. Columnar execution: DuckDB uses vectorized execution with SIMD, while Spark's SQL engine uses row-based Volcano-style iteration.

4. No shuffle overhead: DuckDB runs on a single node with no shuffle; Spark's SQL engine uses sort-based aggregation which spills to disk at 2 cores.

DuckDB's join-aggregate pattern completes each iteration in ~0.5 seconds vs Spark's ~22 seconds.

Python 3.14 Pickle Bug

RecursionError: Stack overflow — Root Cause

Python 3.14 changed its internal stack handling, causing deep recursion in Py4J's pickle serialization of lambda functions. Every RDD operation (map, reduceByKey, groupByKey) serializes the Python function through Py4J, triggering the error. DataFrame operations use column expressions (not lambdas), avoiding the pickle path entirely.

GraphFrames' Python GraphFrame class uses Py4J to delegate to Scala, but constructing the graph from DataFrames and computing out-degrees uses column-based operations that avoid pickle serialization.

Implementation Code

PySpark — Core Iteration (DataFrame API)

contribs = ranks.alias("r") \
    .join(edges.alias("e"), col("r.page") == col("e.src")) \
    .join(out_deg.alias("o"), col("e.src") == col("o.id")) \
    .select(col("e.dst"),
            (col("r.rank") / col("o.outDegree")).alias("contrib"))

totals = contribs.groupBy("dst") \
    .agg(spark_sum("contrib").alias("total_contrib"))

new_ranks = totals.select(
    col("dst").alias("page"),
    (lit(0.15) + lit(0.85) * col("total_contrib")).alias("rank")
)

DuckDB — Core Iteration (SQL)

con.execute("""
    CREATE TABLE contribs AS
    SELECT e.dst, SUM(r.rank / o.cnt) AS total_contrib
    FROM ranks r
    JOIN edges e ON r.page = e.src
    JOIN out_deg o ON r.page = o.src
    GROUP BY e.dst
""")

con.execute("""
    CREATE TABLE new_ranks AS
    SELECT dst AS page, 0.15 + 0.85 * total_contrib AS rank
    FROM contribs
""")

Key Findings

  1. DuckDB is the fastest implementation by a wide margin — 39× faster than PySpark (5.7s vs 224s). For single-node graph analytics, an in-process OLAP engine dramatically outperforms distributed frameworks.
  2. Python 3.14 breaks PySpark RDD/GraphX — The pickle serialization recursion bug makes Python-based RDD operations impossible. The DataFrame API avoids pickle entirely.
  3. GraphFrames from graphframes import GraphFrame works with the 0.8.4 jar — The pip-only v0.6 is incompatible with Spark 3.5; extracting from the Maven jar is required. Once loaded, the Python API works correctly via Py4J.
  4. All three implementations produce identical results — Same top-50 pages (IDs and ranks match exactly), same iteration counts. This validates the algorithm correctness across different frameworks.
  5. GraphFrames and PySpark have equivalent performance — Both run on Spark with similar overhead (~224s). DuckDB's in-process design avoids JVM startup, serialization, and shuffle costs.
