5. Analysis & Observations

Performance Differences

DuckDB is the fastest (7.58s) by a wide margin — 62x faster than custom PySpark and 32x faster than GraphFrames. This is expected because DuckDB uses vectorized columnar execution with minimal orchestration overhead, processing the entire graph in-memory on a single machine without serialization or shuffle costs.

Custom PySpark is the slowest (471.54s) because each iteration requires RDD joins, flatMaps, and reduceByKey operations that trigger shuffles across partitions. On a single machine with only 2 cores, the shuffle/serialization overhead dominates actual computation. However, this implementation reflects the structure that would scale horizontally across a distributed cluster.

GraphFrames (244.45s) sits in the middle. It simplifies implementation to a single API call (g.pageRank()), but still inherits Spark's job scheduling overhead and graph construction costs. The built-in algorithm is optimized within the JVM, avoiding Python serialization overhead, which explains why it's roughly 2x faster than the custom PySpark RDD approach.

Result Consistency

DuckDB and Custom PySpark results are virtually identical — rank values match to approximately 10 decimal places (differences on the order of 10-10 to 10-12). This confirms both implementations follow the same algorithm: initialize ranks to 1.0, propagate rank/degree along edges, apply 0.15 + 0.85 * sum(contributions).

GraphFrames results differ more noticeably. The #1 page (272919) has rank 6649.56 in GraphFrames vs 6531.32 in the other two methods (~1.8% higher). This is because GraphFrames includes all 685,230 vertices (including 4,744 sink nodes with no outgoing links) and redistributes their rank mass via the random teleport mechanism. The custom implementations only track pages with outgoing links, so dangling node mass is effectively lost rather than redistributed. The rank order broadly agrees across all three methods — the same pages appear in the top ranks, just with different absolute values.

Key Takeaways

DuckDB is often fastest for single-machine analytical loops due to vectorized execution and low orchestration overhead.

Custom PySpark can be slower locally because of RDD shuffle/serialization overhead, but it reflects distributed execution structure that would scale on a cluster.

GraphFrames simplifies implementation and can be competitive, though it still inherits Spark job overhead and graph construction costs.

Rank order should broadly agree across methods, with small numeric differences caused by implementation details and floating-point accumulation.
